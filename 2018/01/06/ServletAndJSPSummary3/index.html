Servlet/JSP学习总结（3）Seesion 管理

Posted on 2018/01/06

为什么需要Session管理？

HTTP 协议是无状态的，即不同的请求没有上下文关系，但是很多时候页面上的操作是需要依赖上下文的，比方说页面跳转、登录注册等。这就需要 Cookies 与 Sessions 登场了。

用于保持状态的四种方法有：

  • URL重写
  • 隐藏域（Hidden Fields）
  • cookie
  • HttpSession 对象

URL 重写

工作原理

URL重写的基本原理是直接将一个或多个token作为一个查询字符串（Query String）添加到一个URL中。格式如

url?key-1=value-1&key-2=value-2

URL与token之间用一个问号?隔开，两个token之间则用一个&符号。

缺点

  • 一些Web浏览器对URL字符的大写有限制，插入的token是有限，能够携带的信息也是有限的。
  • 某些token若携带了特殊的字符，如空格、&和?等。此时需要对token进行编码。
  • token中的信息完全暴露在URL中，对于一些敏感信息就不适应了。

适用场景

针对上述的缺点，URL重写适合如下的使用场景：

  • 需要保持状态，传递信息的安全性要求不高。
  • 不需要跨越太多页面

Hidden Fields

工作原理

与URL重写方式类似，但它不是将信息添加到URL内，而是将它们放在HTML表单的隐藏域中，当用户提交表单时，隐藏域的值也传送到服务器中。

优缺点

  • 只适用于那些包含表单的页面，此时才能使用隐藏域。
  • 当需要跨越很多页面时，实现就变得困难。
  • 相对于URL重写，能携带更多的信息，并且不需要对字符进行编码。

Cookies

Cookie 是自动地在Web服务器和浏览器之间来回传递的一小块信息。适用于那些需要跨越许多页面的信息。

Cookie 嵌入在HTTP Header信息中，是HTTP头中的一个字段，传输过程中由HTTP协议处理。

浏览器对Cookie的存储有限制，例如Chrome浏览器对每个域名最多只支持50个Cookie，FirFox最多支持4097个字的Cookie。

用户可以在浏览器内拒绝接受cookie。

每次客户端的访问都必须传回Cookie，后面Session的出现解决了这个问题。

向客户端发送Cookie

一般过程：

Servlet使用Cookie的构造函数，传入指定的名称和值，创建一个或多个Cookie。通过Cookie.setXxx()方法为每个Cookie设置属性值。最后使用HttpServletResponse.addCookie(Cookie c)方法将cookie插入HTTP响应Header中。

下面是一个具体的例子：

1. 创建Cookie对象

创建一个用户ID的cookie。

Cookie userCookie = new Cookie("userId","1234");

注： 构造方法接受两个字符串：cookie的名称和值。两个字符串不可包含空格和[],?,=,@,:,;字符。

2. 为Cookie设置MaxAge

设置c的时效为24小时，服务端将保留这个cookie24小时。

userCookie.setMaxAge(60*60*24);

setMaxAge()接受一个int值表示cookie的时效，单位为秒。若不设置MaxAge属性，默认情况下为-1

3. 使用HttpResponse 添加Cookie

为了将以个Cookie发送到浏览器，需要在HttpResponse上调用add方法：

response.addCookie(userCookie);

读取来自客户端的Cookie

读取来自客户端的Cookie主要操作有：

  • 在HttpRequest上调用getCookies()方法，得到一个Cookie[].
  • 对Cookie[]进行遍历，调用每个cookie的getName()方法，直到找到目标cookie为止。然后一般调用getValue()得到目标值。

Cookie类的其他方法

在将cookie添加到HttpResponse前，可以使用setXxx方法设置cookie的各种属性，Xxx是希望设置的属性名称。

属性是从服务器发送到浏览器的报头的一部分，它不属于由浏览器返回服务器的报头，也就是说来自于浏览器的cookie并没有设置这些属性。

下面是常用的cookie属性方法：

  • public String getName()
    该方法读取cookie的名称，通常与public String getValue()搭配使用。

  • public String getValue()，public void setValue(String cookieValue)
    前着读取与该Cookie相关的值，后者设置该Cookie的值。

  • public void setMaxAge(int lifetime)，public int maxAge()
    前者设置Cookie的最大时效，lifetime单位为秒，默认值为负值，表示仅仅用于与当前浏览器的会话，当用户退出浏览器时Cookie失效。后者返回该Cookie的最大时效。Cookie不提供删除方法，只能使用setMaxAge()设置maxAge为0，然后添加到response中，达到删除的效果。

  • public String getPath()
    设置或取得Cookie所使用的路径，若没有指定路径，浏览器只会将该Cookie返回给发送Cookie的页面所在目录或之下的URL。例如，若服务器从
    http://localhost/toys/Cars/showCar.html发送Cookie，那么浏览器连接到http://localhost/toys/bikes/showBikes.html时会返回Cookie给服务器，而访问http://localhost/cds/index.html时不会返回Cookie。

Session

Session技术是建立在Cookie或URL基础之上的，将用于客户端与服务端来回交互的信息以形如<属性名，属性值>的键值对的形式，添加进称为HttpSession对象的散列表，这里的属性值可以是任意的Java对象，这比Cookie对值的类型限制只能是字符串类型要灵活的多。

在Servlet中使用Seesion一般涉及以下4个操作：

获取当前请求的HttpSession对象

调用HttpServletRequest.getSession()得到当前请求的HttpSession对象，若该请求不存在HttpSession对象，则会在服务端创建与一个新的属于该请求的HttpSession对象，返回，getSession(true)有着相同的效果。若不希望服务端创建新的HttpSession对象，可以调用getSession(false)方法，返回null。

在HttpSession中查找和获取交互信息

调用httpSession.getAtrribute()，得到返回值并将返回值转换为特定的对象。这里的返回值不一定是String，可以是任意Java对象，因此需要对返回值做转换。
HttpSession对象存在于服务器，它们并不在网络上来回传送，只通过底层的机制，如Cookie或URL重写，自动与客户端的用户关联起来。

为HttpSession对象添加信息与更新信息

调用HttpSession.setAtrribute(String name,Object value)，添加一个名称name,类型为value的信息。其中Object value需要实现Serializable接口。

同样是更新操作，对于HttpSession对象存放的属性值是Inmutable（不可修改）类，还是Mutable（可修改）类，这两种的情况的处理方式是不同的。
